# Incident Report: 2026-05-27 delayed Discord alert

## What happened

- 2026-05-26 오후 5:58(KST) 무렵 `alex-notes` 저장소에 `2026-05-26.md`가 올라왔습니다.
- 그런데 Discord 알림은 즉시 가지 않았고, 다음 성공 실행인 2026-05-27 오전 1:29(KST)에 도착했습니다.

## Why the alert was delayed

이 프로젝트는 파일의 "업로드 시각" 자체를 기준으로 알림을 보내지 않습니다. 대신 이전에 성공적으로 저장한 `lastCheckedSha`와 최신 커밋 SHA를 비교해서, 그 사이에 새로 추가된 `.md` 파일이 있는지 확인합니다.

이번 지연의 직접 원인은 2026-05-26 오후 9:18(KST) 스케줄 실행 실패였습니다.

- GitHub Actions run `Check Alex Notes #44`가 실패했습니다.
- 실패 위치는 워크플로 마지막의 `git push` 단계였습니다.
- 에러 메시지는 `unable to access 'https://github.com/Subin9227/git-alarm/': The requested URL returned error: 403` 이었습니다.

이 실패 때문에 러너 내부에서 갱신한 `data/state.json`이 저장소에 push되지 못했습니다. 그 결과 다음 실행은 이전 상태를 그대로 읽었고, 이미 올라와 있던 `2026-05-26.md`를 "아직 처리하지 않은 새 파일"로 다시 판단했습니다. 그래서 2026-05-27 오전 1:29(KST)에 Discord 알림이 전송되었습니다.

## What was the problem

- 워크플로는 상태 파일 `data/state.json`을 커밋하고 push해야 다음 실행에서 중복/지연 알림을 막을 수 있습니다.
- 하지만 저장소의 GitHub Actions 기본 권한이 쓰기 권한 없이 설정되어 있었기 때문에 `git push`가 403으로 막혔습니다.

즉, 문제의 핵심은 알림 로직이 아니라 GitHub Actions 권한 설정이었습니다.

## What was changed

다음 설정을 수정했습니다.

- `Settings > Actions > General > Workflow permissions`
- `Read repository contents permission`에서 `Read and write permissions`로 변경

이 변경 이후 워크플로가 `data/state.json`을 정상적으로 커밋/push할 수 있게 되었고, 이후 실행은 정상적으로 상태를 이어받게 됩니다.

## Result

- 수동 실행 `Check Alex Notes #52`가 성공했습니다.
- 이후 새 `.md` 파일이 올라왔을 때 Discord 알림도 다시 정상적으로 동작했습니다.
- 이제는 매번 `Run workflow`를 누를 필요 없이 스케줄 실행이 자동으로 체크합니다.

## Follow-up

- 이 워크플로는 매시간이 아니라 30분마다 실행됩니다. 현재 cron은 매시 `:07`, `:37`입니다.
- GitHub Actions의 `schedule` 실행 시각은 항상 정확히 맞아떨어지지 않을 수 있습니다. GitHub는 scheduled workflow를 기본적으로 UTC 기준으로 처리하고, Actions 부하가 높은 시간대에는 실행이 몇 분에서 더 길게 지연될 수 있다고 안내합니다. 그래서 실행 목록에는 `:07`, `:37` 대신 `21:18`, `01:28`처럼 보이는 시각이 남을 수 있습니다.
- `README.md`의 실행 주기도 실제 동작에 맞게 수정했습니다.
- 앞으로 빨간 실패가 보이면 먼저 `Commit state changes` 단계와 `git push` 권한 에러 여부를 확인합니다.
